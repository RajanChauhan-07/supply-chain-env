System Overview

What this backend is built to do

Purpose

The backend simulates a real supply chain operations workflow. An agent acts like an operations manager handling delays, supplier failures, order risk, budget pressure, and escalation decisions.

Core Stack

The service is built with FastAPI and typed Pydantic models. It exposes `reset`, `step`, `state`, `tasks`, `grade`, and `validate` endpoints, plus a root health check for deployment verification.

Agent Inputs

Every step returns disruptions, orders, suppliers, budget, metrics, and a status message. In the hard task, some supplier reliability data stays hidden until the agent explicitly investigates it.

Agent Outputs

The agent responds with one structured action such as reroute, substitute, delay, cancel, escalate, or investigate. The backend validates the action, updates state, and returns reward plus progress.

Backend Flow

How the environment works from start to finish

01

Health check confirms the service is alive

The root endpoint returns metadata about the environment and all major routes. This is the first signal that the deployment is healthy and reachable.

02

`reset()` starts a fresh episode

A chosen task such as `task_easy`, `task_medium`, or `task_hard` loads its initial suppliers, disruptions, orders, budget, and scoring context.

03

The observation becomes the agent's decision context

The agent sees active disruptions, at-risk orders, alternative suppliers, budget remaining, and running metrics like current score and saved orders.

04

`step(action)` applies one operational decision

The backend validates the action model, mutates the environment state, computes immediate reward, and decides whether the episode should continue.

05

Reward shaping drives meaningful progress

Good reroutes, correct escalations, useful investigations, and revenue protection improve the score. Invalid, redundant, late, or costly decisions reduce it.

06

`state()` exposes the full internal episode snapshot

This lets a caller inspect the current environment state without advancing the episode, which is useful for debugging and validation.

07

`grade()` converts the run into a deterministic final score

The grader returns a normalized score in the range `[0.0, 1.0]` plus a breakdown of what contributed to success or failure.

Task Ladder

Three tasks with increasing operational pressure

Easy

Single Lane Disruption

One important order is threatened by a single supplier issue. The agent needs to pick a viable alternative and preserve value efficiently.

  • Max steps: 10
  • Pass threshold: 0.60
  • Expected range: 0.80 to 1.00

Medium

Multi-Point Failure

Several disruptions and budget limits force prioritization. The agent cannot save everything equally and must make tradeoffs intentionally.

  • Max steps: 20
  • Pass threshold: 0.45
  • Expected range: 0.70 to 0.90

Hard

Cascade Crisis

Multiple high-severity disruptions, tight budget, hidden supplier risk, and partial infeasibility make the final task a realistic crisis-management test. Some supplier reliability is intentionally hidden until investigated.

  • Max steps: 30
  • Pass threshold: 0.30
  • Expected range: 0.45 to 0.70

API Surface

What each backend route is responsible for

GET /

Returns deployment metadata and confirms the environment is healthy.

POST /reset

Starts a fresh episode for a selected task and returns the initial observation.

POST /step

Accepts one typed action and returns updated observation, reward, done, and info.

GET /state

Returns the full internal state without advancing the episode.

GET /tasks

Lists all tasks, difficulty levels, and metadata.

GET /grade

Returns the deterministic final score and grading breakdown for the episode.

GET /validate

Runs environment self-checks to confirm the system behaves correctly.

How To Read The Backend

A systematic map of the codebase

1. API entrypoint

Start with backend/app/main.py. This file defines the FastAPI app, CORS setup, route handlers, and validation endpoints.

2. Configuration

Open backend/app/config.py to see how app metadata and environment variables are configured.

3. Engine

The environment engine lives in backend/app/environment/engine.py. This is where state transitions, action handling, and episode flow are coordinated.

4. Task definitions

Task scenarios are split across backend/app/tasks/. Each task defines the starting world, task difficulty, and expected objective pressure.

5. Typed models

Pydantic schemas in backend/app/models/ define the Observation, Action, Reward, and state contracts used by the API.

6. Graders

Grading logic in backend/app/graders/ converts the final episode outcome into deterministic normalized scores.

Run Locally

Step-by-step instructions

Step 1

Install the backend dependencies.

pip install -r backend/requirements.txt

Step 2

Start the FastAPI server on port 7860.

uvicorn backend.app.main:app --host 0.0.0.0 --port 7860

Step 3

Confirm the backend is responding.

curl http://localhost:7860/
curl http://localhost:7860/tasks
curl http://localhost:7860/validate

Step 4

Set inference environment variables for the OpenAI-compatible client.

API_BASE_URL=<provider endpoint>
MODEL_NAME=<model id>
OPENAI_API_KEY=<api key>
ENV_URL=http://localhost:7860

Step 5

Run the baseline inference script.

python inference.py
# or for pure model-only evaluation:
STRICT_BASELINE=1 python inference.py

Step 6

Run the backend tests to confirm everything stays healthy.

python -m unittest discover -s backend/tests -p "test_*.py"

Reward Design

What the backend encourages and what it penalizes

Positive signals

  • Fulfilling orders on time
  • Choosing cost-efficient, reliable suppliers
  • Escalating critical disruptions correctly
  • Running useful investigations before risky decisions
  • Preserving high-value revenue under budget

Negative signals

  • Missing deadlines or losing orders
  • Overspending the budget
  • Using unreliable suppliers
  • Repeating redundant actions
  • Submitting invalid or harmful actions

Evaluation Notes

How to interpret baseline and judge runs

Deterministic environment

The environment itself is deterministic. Differences in score should come from agent decisions, not random world changes between runs.

Baseline modes

The baseline script reports whether each task was model-only, hybrid, or fallback-only. `STRICT_BASELINE=1` disables fallback rescue for pure model runs.

What hard task measures

The hard task is not just about rerouting. It checks escalation discipline, budget tradeoffs, and whether the agent investigates hidden supplier risk.

What a good score means

Strong agents should solve easy reliably, perform well but not perfectly on medium, and still face meaningful tradeoffs on hard.
